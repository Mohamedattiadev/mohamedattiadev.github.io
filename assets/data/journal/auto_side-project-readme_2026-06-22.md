# How I structure a side-project README

A good README is a sales page for your project. Make it skimmable. People decide in 15 seconds.

My template, in order:

1. **One sentence**, top of the file. What the project is, in plain English.
2. **A picture or GIF.** If the project has a UI, show it. People believe pictures.
3. **Three bullet points.** The things it does, the things it does *not* do.
4. **Install + run.** Two or three commands max. Copy-paste ready.
5. **One example.** Real input, real output.
6. **A link to "why".** A blog post or short doc explaining the motivation. Optional but powerful.

I never write a long "Features" list. I never explain my life story. I never use words like *robust* or *enterprise-grade*.

A good README respects the reader's time. People who can read it in 30 seconds become users. People who cannot, leave. That is the whole game.

If you have one hour for your project this week, spend it on the README.
